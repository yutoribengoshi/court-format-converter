## 40分の説教

新人弁護士の頃、起案した準備書面を上司に見せた（K先生ありがとうございます）。

内容についてのフィードバックはほぼなかった。代わりに40分間、インデントの話をされた。

「第１の１と、単独の１の位置が揃ってない」「本文の頭が見出しの頭と揃ってない」「ここ半角になってる」「ここのフォントが違う」

正直、その40分で書き直せるのではと思った。

でも先輩が言いたかったのは「書式が崩れた書面は裁判官に読んでもらえない」ということだった。裁判官は1日に何十通も書面を読む。書式が統一されていない書面は、それだけで読む気が失せる。

理屈はわかる。でも理不尽だとも思った。

## 書式整形という苦行

弁護士の書面作成で最も非生産的な作業は書式整形だ。

- ページ設定（A4、余白、行数、文字数）
- フォント統一（MS 明朝 12pt）
- 見出し番号体系（第１→１→(1)→ア→...）
- インデント（込み出し、ぶら下げ）
- 半角→全角変換

内容とは一切関係ない。しかし間違えると40分説教される。

既存のツールとして「岡口マクロ」というWord VBAマクロがある。裁判所書式の事実上の標準ツールで、ほぼ全ての弁護士が知っている。だが2020年で更新が止まっており、完全な機能はWindowsでしか動かない。Macを使う弁護士は一部、手作業で整形するしかなかった。

## 作ったもの

文化審議会建議「公用文作成の考え方」（令和4年）および裁判所実務の書式慣行に基づいて、docxファイルの書式を一括整形するツールを作った。

**court-format-converter**
https://github.com/yutoribengoshi/court-format-converter

Python CLI版とWordアドイン版の2通りで使える。

### Python CLI版

```bash
pip install python-docx
python3 court_format_converter.py 準備書面.docx
```

これだけ。出力ファイルが裁判所書式に整形されて出てくる。

### Wordアドイン版

Wordのリボンに「裁判所書式」ボタンを追加。開いている文書をその場で整形する。Python不要。Windows/Mac両対応。

GitHub Pagesでホストしているので、manifest.xmlをダウンロードしてWordのアドインフォルダに入れるだけで使える。

## 機能

### ページ設定

A4縦、余白（上35mm/下25mm/左30mm/右20mm）、26行×37文字のグリッド設定、フッターにページ番号。

### フォント統一

日本語はMS 明朝 12pt、英数字はTimes New Roman 12pt。テーブル内は10pt。タイトル行（準備書面、訴状等）は自動検出して16pt太字。

### 見出し自動検出とインデント

テキストパターンから7階層の見出しを自動判定。ぶら下げインデントで番号を込み出し、本文の頭を見出しタイトルの頭と揃える。

| レベル | 番号形式 | 例 |
|---|---|---|
| 1 | 第＋全角数字 | 第１、第２ |
| 2 | 全角数字 | １、２ |
| 3 | 半角括弧＋数字 | (1)、(2) |
| 4 | 全角カタカナ | ア、イ |
| 5 | 半角括弧＋カタカナ | (ｱ)、(ｲ) |
| 6 | 全角小文字 | ａ、ｂ |
| 7 | 半角括弧＋小文字 | (a)、(b) |

先輩に怒られた「第１の１と単独の１の位置」は自動で揃う。

### 見出し番号の自動再付番

これが一番の売り。

文書の番号体系が裁判所書式と異なる場合、自動的に正しい番号に変換する。

```
変換前:                          変換後:
１　被告の主張            →    第１　被告の主張
(1)　契約締結の経緯       →    １　契約締結の経緯
ア　代金の支払            →    (1)　代金の支払
```

番号体系のレベルを自動検出し、最上位の番号を第１に、以下を順にシフトする。カウンターは上位レベルが進むと自動リセットされる。

括弧付き数字の特殊文字（⑴⑵⑶）も通常の(1)(2)(3)の表記揺れとして検出・修正する。

### その他

- 半角→全角変換（数字・英字・カタカナ・括弧・記号）
- 元文書の手動全角スペースインデントを除去
- 箇条書き（１．等）のぶら下げインデント
- テーブルはフォント10pt化＋セル余白最小化＋列幅自動調整

## 技術的に面白かった話

### インデントの地獄

裁判所書式のインデントは「込み出し」という独特のルールがある。見出し番号が左に飛び出して、本文の頭が見出しのタイトルと揃う。

```
第１　走行履歴の出典
　　　　被疑者の父親所有の車両に搭載された...    ← 1行目
　　　履歴画面のスクリーンショット...              ← 2行目は「走」と揃う
```

python-docxでは`w:left`と`w:hanging`をtwips（1/20ポイント）で直接指定することで実現した。グリッドベースの`w:leftChars`を使うとWordの内部計算と実際の文字幅がズレる。

### Wordアドインの闇

Office.jsでも同じ整形をしたかったが、Office.jsの`paragraph.leftIndent`は`w:leftChars`属性を削除しない。古い変換結果の残骸と新しい設定が競合して、インデントが1文字ズレる問題が発生した。

解決策: Office.jsのプロパティAPIを使わず、`body.getOoxml()`でOOXMLを取得→DOMParserで解析→DOM操作で変換→`body.insertOoxml()`で書き戻し。OOXML直接操作なら`w:ind`要素を丸ごと削除して再設定できる。

テーブル内段落の分離も、`hasAncestor(paragraph, 'tbl')`でDOMツリーを歩いて判定した。Office.jsの`parentTableOrNullObject`はロードでハングする。

## 今後の予定

LLMによる以下の機能を開発中。

- **AI構造化**: 見出し番号のない文書や、番号体系が完全に崩れた文書にLLMが見出しを自動付与
- **連番ミス修正**: 「１→２→⑶」のようなレベルまたぎのヒューマンエラーをLLMが文脈から判断して修正
- **公用文変換**: 文体を公用文の作法に沿って自動整形
- **陳述書ドラフト生成**: 打ち合わせメモから裁判所提出用の陳述書を自動生成

これらのAI機能はLLMのAPI利用料が発生するため、有料プランでの提供を予定している。書式整形機能は引き続き無料。
